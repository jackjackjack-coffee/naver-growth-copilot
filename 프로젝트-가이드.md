## 1. 한 줄 요약

**"네이버쇼핑 셀러 컨설턴트가 매주 '어떤 셀러를 먼저 챙겨야 하는가'를 빠르게 판단하고, 그 근거를 담은 AI 주간 브리프를 자동으로 만들어 주는 영업관리 도구."**

- 입력: 셀러별 성과 지표(GMV·성장률·전환율·리뷰·광고 효율 등)
- 처리: 점수화 → 우선순위 정렬 → 근거 요약
- 출력: 대시보드 + AI 주간 브리프 + 현장 검증 계획

## 5. 데이터 모델 — 셀러 1명은 이렇게 생겼다

`app.js`의 `sellers` 배열에 있는 객체 하나의 의미:

```js
{ id:"BBL", name:"블룸랩", category:"beauty",
  gmv:1210, growth:-4.8, health:43, conversion:2.1,
  review:4.1, adRoas:1.8, risk:"Action",
  action:"상세페이지 표현 정비 및 할인 누수 차단", stage:"브리프 발송" }
```

| 필드 | 의미 | 단위/값 |
| --- | --- | --- |
| `id` | 셀러 약어 코드 (산점도 버블 라벨) | 예: BBL |
| `name` | 셀러명 | 한글 |
| `category` | 카테고리 **내부 키** | beauty/fashion/sports/living/grocery |
| `gmv` | 거래액 | **백만원** (1210 = 12.1억) |
| `growth` | 최근 성장률 | % |
| `health` | 셀러 헬스(종합 건강도) | 0~100 |
| `conversion` | 전환율 | % |
| `review` | 리뷰 점수 | 5점 만점 |
| `adRoas` | 광고 ROAS(광고비 대비 매출) | 배수(x) |
| `risk` | 리스크 등급 **내부 키** | Low(안정)/Watch(주의)/Action(조치) |
| `action` | 추천 액션 | 한글 |
| `stage` | 컨설팅 퍼널 단계 | 데이터 검토 / 브리프 발송 / 컨설팅 대기 / 액션 수락 |

> **내부 키 ↔ 한글 라벨**: 코드 안에서는 `beauty`, `Low` 같은 영어 키로 다루고, 화면에는 `categoryLabel()`·`riskLabel()` 함수가 `뷰티`, `안정`으로 바꿔서 보여줍니다. (필터·검색은 키로, 표시는 한글로 → 둘 다 깔끔)

> 📐 **"이 숫자들은 어떻게 정한 거예요?"** → 각 지표의 현실적 범위·임계값·가정 근거는 별도 문서 **[데이터-가정.md](데이터-가정.md)** 에 지표별로 정리했습니다. (면접에서 "왜 이 값?" 질문 방어용)

## 6. 화면(섹션)별 의미와 로직

### 6-1. 대시보드 — "이번 주 전체 그림"

**상단 KPI 카드 4개**

| 카드 | 계산 로직 | 해석 |
| --- | --- | --- |
| 관리 GMV | 모든 셀러 `gmv` 합계 ÷ 100 → 억 | 내가 관리하는 매출 규모 |
| 평균 전환율 | `conversion` 평균 | 유입이 주문으로 얼마나 잘 이어지나 |
| AI 브리프 대기 | `risk === "Action"` 인 셀러 수 | 지금 당장 손봐야 할 셀러 수 |
| 컨설팅 헬스 | `health` 평균(반올림) | 담당 포트폴리오 전체 건강도 |

> 단위 변환 팁: `gmv`는 **백만원**이라 100으로 나누면 **억**이 됩니다 (1억 = 100백만원). 그래서 `formatGmv()`는 `값/100 + "억"`.

**셀러 기회 맵 (산점도)** — 한 장으로 우선순위를 보는 핵심 시각화

- **가로축(X) = 셀러 헬스(0~100)** : 오른쪽일수록 건강함
- **세로축(Y) = 성장률(약 -8%~+24%)** : 위쪽일수록 성장 중
- **버블 크기 = GMV** : 클수록 매출 규모가 큼
- **버블 색 = 리스크** : 🟢안정 🟠주의 🔴조치

좌표 계산식(요지):
```
x = 8 + (health/100) * 84      // 헬스 0~100 → 화면 가로 8%~92%
y = 12 + ((growth+8)/32) * 76  // 성장률 -8%~+24% → 화면 세로 12%~88%
size = 34 + min(gmv/45, 48)    // GMV가 클수록 버블이 커지되 최대 크기 제한
```

**해석법:**
- **오른쪽 위 + 큰 초록 버블** = 잘하고 있는 큰 셀러 → 확장 기회
- **왼쪽 아래 + 빨강 버블** = 작고 아픈 셀러 → **가장 시급**
- 버블을 클릭하면 해당 셀러 상세로 이동합니다.

**다음 추천 액션 (우선순위 큐)**
- 정렬 로직: `health` **오름차순**(아픈 셀러 먼저), 동점이면 `gmv` **내림차순**(규모 큰 셀러 먼저) → 상위 4개
- 의미: "가장 건강이 나쁘면서 매출 영향도 큰 셀러"를 자동으로 맨 위에 올림

**카테고리별 GMV (막대)**
- 카테고리별 `gmv` 합계, 막대 길이 = `값 / 최댓값`
- 의미: 어떤 카테고리가 내 포트폴리오 매출을 끌고 있나

**셀러 컨설팅 퍼널 (4단계)**
- `데이터 검토 → 브리프 발송 → 컨설팅 대기 → 액션 수락` 각 단계에 몇 명이 있는지 카운트
- 의미: 컨설팅이 어느 단계에서 막혀 있는지(병목)를 보여주는 운영 지표

### 6-3. AI 리포트 — "근거 있는 주간 브리프 자동 생성" (이 프로젝트의 하이라이트)

브리프를 만드는 **두 가지 경로**가 있습니다.

**(A) 템플릿 기반 — 항상 작동(오프라인, 키 불필요)**
- `reportMarkdown()` 함수가 규칙으로 글을 조립
- **핵심 진단 분기(rootCause):**
  - 전환율 < 2.5% → "유입이 주문으로 안 이어짐"
  - 아니고 ROAS < 2.5x → "광고 효율이 낮음"
  - 둘 다 아니면 → "성장세 양호, 확장 테스트 여력"
- **어조(tone) 3종:** 임원 보고용 / 현장 컨설턴트용 / 셀러 대상 → 같은 데이터, 다른 말투

**(B) AI 생성 — Google Gemini 호출**
- 흐름: `AI로 생성` 버튼 → 키 확인 → `buildPrompt()`로 한국어 프롬프트 작성 → Gemini 호출 → 결과를 화면에 렌더링
- 프롬프트 구성(요지): **역할(네이버쇼핑 셀러 컨설팅 애널리스트) + 작성 규칙(형식·어조·근거 인용) + 셀러 데이터**
- 실패하거나 키가 없으면 → **자동으로 템플릿으로 폴백**하고 오류 메시지 표시 (절대 빈 화면이 안 됨)

**API 키 보안 설계 (중요 포인트):**
- 키를 **코드/저장소에 절대 저장하지 않음** → 정적 사이트로 배포해도 노출 없음
- 입력한 키는 브라우저 `localStorage`에만 저장, 요청은 브라우저 → Google로 직접
- 저장 후 입력란은 비워서 화면에도 남기지 않음
- 면접 어필 포인트: "비밀키를 하드코딩하지 않는다"는 기본기를 보여줌

**재사용 워크플로우(프롬프트 체인) 5단계** — 실제 운영 시 데이터가 흐르는 순서를 문서화:
`테이블 결합 → 점수화 → 원인 요약 → 브리프 생성 → 액션 로깅`

### 6-5. SQL 로직 — "데이터로 우선순위를 정하는 방법"

**🟢 쉽게 말하면 (이것만 알면 됨):** 이 쿼리는 세 가지를 합니다.
1. **주간 데이터 수집** — 각 셀러의 최근 8주 지표(매출·전환·리뷰·광고 효율)를 모음
2. **기회 점수로 압축** — 그 지표들을 하나의 '기회 점수'로 변환 (지표가 낮을수록 점수↑ = 개선 여지가 큼)
3. **우선순위 정렬** — 점수가 높은 순서로 줄 세워 "누구부터 챙길지" 출력

> 화면에 진짜 SQL을 띄운 이유는 공고의 *(우대) SQL·데이터 분석 경험*에 대응하기 위해서입니다. 비즈니스 리뷰어(그리고 본인)는 위 3줄만 이해하면 충분하고, SQL 문법 자체를 읽을 필요는 없습니다.

---

**🔧 더 자세히 (기술 디테일 — 읽고 싶을 때만):** 화면의 쿼리는 **BigQuery 표준 SQL**이며, 두 단계(CTE)로 구성됩니다.

**① `seller_week` (주차별 집계)**
- 최근 8주 데이터를 셀러·주차별로 모음
- 전환율 = 주문수 / 세션수, ROAS = 광고매출 / 광고비, 리뷰 평균
- `LAG(..., 4)` = **4주 전 GMV**를 같은 행으로 끌어와 성장률 계산용으로 준비

**② `scored` (점수 산출)**
```
opportunity_score =
    0.35 * (1 - PERCENT_RANK(GMV))
  + 0.20 * (1 - PERCENT_RANK(전환율))
  + 0.15 * (1 - PERCENT_RANK(리뷰))
  + 0.15 * (1 - PERCENT_RANK(ROAS))
  + 0.15 * 규모 보정
```
- `PERCENT_RANK()` = 그 값이 전체에서 몇 % 위치인지(0~1)
- **`1 - PERCENT_RANK`** 를 쓰는 이유: 값이 **낮을수록 점수↑** → "지금 도움이 필요한 / 개선 기회가 큰" 셀러가 위로 올라옴
- 마지막에 `ORDER BY opportunity_score DESC LIMIT 50` → 상위 50명만

**가중치 패널 (오른쪽)**: 성장 갭 35% · 전환율 하락 20% · 리뷰 리스크 15% · 광고 효율 15% · 컨설팅 준비도 15%

> 🔎 **정직한 해석(면접에서 쓸 수 있는 디테일):**
> 데모 SQL에서 35% 가중치는 단순화를 위해 **GMV 백분위**에 걸려 있습니다(규모 큰 셀러가 약간 우선되는 효과). UI 라벨은 이를 '성장 갭'으로 개념화한 것이라, 엄밀히는 `growth_rate`(4주 전 대비)를 점수에 **직접** 넣는 것이 더 정확합니다.
> → 즉 "현재는 데모용 단순 모델이고, 실데이터 연결 시 성장률을 직접 반영해 고도화하겠다"가 자연스러운 개선 스토리입니다.

### 6-6. 사용성 테스트 — "현장에서 검증하는 계획"

- **테스트 과제 6개:** 각 과제마다 *방법*과 *성공 기준*을 명시
  - 예) "긴급 셀러 상위 3곳 찾기" → 컨설턴트 대상 시간 측정 → **90초 이내**
- **도입 성공 신호 4개:** 브리프 작성 5분 이내 · 액션 수락률 60%+ · 근거 신뢰도 4.2/5 · 주 3회 재사용
- 의미: "만들고 끝"이 아니라 **측정 가능한 기준으로 검증**한다는 프로덕트 사고를 보여줌

## 7. 알아두면 좋은 핵심 함수 (app.js)

> ⚙️ **구조 업데이트(2지망 연계 후):** 엔진을 **도메인 설정(`DOMAINS.seller` / `DOMAINS.creator`)** 으로 일반화해, 같은 렌더 함수가 모드만 바꿔 끼웁니다. 아래 함수표는 그 일반화된 이름입니다. (셀러 단독 동작은 그대로 보존)

| 함수 | 하는 일 |
| --- | --- |
| `domain()` | 현재 모드(`state.mode`)의 도메인 설정 객체 반환 — 라벨·점수·포맷·프롬프트의 단일 출처 |
| `filteredItems()` | 카테고리 필터 + 검색어로 현재 도메인 목록 거름 (둘 다 AND) |
| `domain().formatPrimary(v)` | 셀러=억(`v/100`), 창작자=만회/천회 변환 |
| `domain().riskColorOf / riskLabels`, `categoryLabel` | 내부 키 → 색/한글 라벨 매핑 (도메인별) |
| `domain().template(it, tone)` | 템플릿 기반 한국어 브리프 생성(규칙) — 셀러/창작자 각각 |
| `domain().prompt(it, tone)` | Gemini용 한국어 프롬프트 조립 — 셀러/창작자 각각 |
| `generateWithGemini()` | Gemini REST 호출 + 오류 처리 (도메인 공통) |
| `mdToHtml()` | 제한적 마크다운(`###`, `**굵게**`, 빈 줄 단락) → HTML |
| `setMode(mode)` | 셀러 ↔ 창작자 토글 — 라벨 재적용 + 전체 재렌더 |
| `applyDomainLabels()` | 화면 라벨(브랜드·탭·표 헤더·어조 옵션 등)을 현재 도메인에 맞게 교체 |
| `renderFlywheel()` | 커머스-콘텐츠 플라이휠(주제→태그→셀러 기회) + NCP 매핑 렌더 |
| `exportCsv()` | 현재 목록을 CSV로 내보냄. **UTF-8 BOM**을 붙여 엑셀에서 한글 안 깨지게 함 |

## 8. 한계와 다음 단계 (솔직한 평가)

**지금의 한계**
- 데이터는 **합성(synthetic)** — 실제 NAVER 데이터가 아님
- AI 출력은 검증 전 (환각 가능) → 그래서 "사람이 수정"하는 설계
- SQL 점수 모델은 **데모용 단순화** (위 6-5 해석 참고)

**다음 단계(고도화 아이디어)**
1. 실데이터/시트 업로드 연결, `growth_rate`를 점수에 직접 반영
2. AI 브리프에 대한 컨설턴트 수정·수락 **로깅** → 사용성 테스트 데이터로 축적
3. 셀러 대상 발송용 톤 분리, 발송 이력 관리
4. 벤치마크 보드를 1페이지 케이스 스터디 PDF로 출력

## 10. 2지망 연계 — "한 엔진, 두 사용자" (클립 창작자 모드)

> 이 도구는 1지망(셀러 컨설팅)에서 출발해, **같은 엔진으로 2지망 「클립 창작자 콘텐츠 인사이트 리포트 자동화」까지** 보여주도록 확장했습니다.

### 10-1. 왜 한 엔진으로 묶이나
코파일럿의 핵심은 **수집 → 점수화 → 우선순위 → 근거 → AI 브리프 → 검증** 파이프라인입니다.
대상이 *셀러*에서 *클립 주제*로 바뀔 뿐, "데이터를 근거 있는 추천으로 바꾼다"는 엔진은 동일합니다.
그래서 코드도 새 엔진을 만들지 않고, **도메인 설정(`DOMAINS.seller`/`DOMAINS.creator`)** 만 바꿔 끼웁니다.

### 10-2. 모드 토글로 바뀌는 것
| 화면 | 셀러 모드 | 창작자 모드 |
| --- | --- | --- |
| 대시보드 | 셀러 기회 맵(준비도×성장률) | **주제 기회 맵(낮은 포화도×성장 속도)** |
| 다음 액션 | 가장 도움 필요한 셀러 | **이번 주 만들 주제 3** |
| AI 리포트 | 주간 셀러 브리프 | **주간 창작자 브리프**(추천 주제+왜 떴나+훅+커머스 연결) |
| SQL | 셀러 기회 점수 | **주제 기회 점수**(성장35·저포화25·검색15·재생산15·태그전환10) |
| 벤치마크 | 영업관리 도구 6종 | **창작자 분석 도구 6종**(크리에이터 어드바이저·YT Studio·TikTok…) |
| 사용성 | 컨설턴트 검증 | **창작자 검증**(리포트 보고 실제 제작했나) |

> 점수 방향이 도메인별로 다릅니다: 셀러는 **저성과=시급**(도움 필요), 창작자는 **고성장·저포화=기회**(지금 들어가면 좋음). `priorityCompare`가 이를 분기합니다.

### 10-3. 플라이휠 화면 — 1지망 ↔ 2지망을 한 화면에서 잇기
클립의 **정보 태그**(쇼핑 태그 전년比 약 +350%)와 **쇼핑 커넥트** 때문에 창작자 콘텐츠는 곧 셀러 수요로 이어집니다.
플라이휠 탭은 이 연결을 시각화합니다: **뜨는 클립 주제 → 정보 태그 → 네이버쇼핑 수요 신호 → 셀러 기회 → 셀러·콘텐츠 성장.**
또 카테고리별로 *가장 뜨는 주제*를 *같은 카테고리에서 가장 도움이 필요한 셀러*에 직접 연결해 보여줍니다(같은 합성 데이터 두 벌을 카테고리로 매칭).

### 10-4. NCP 스택 매핑 (실서비스 가정)
플라이휠 하단에 **실서비스라면 어떤 NAVER Cloud 부품으로 구현되는지**를 1:1로 적어 뒀습니다 —
Media Intelligence(영상 멀티모달→신호) · NCLUE(쇼핑 인텐트) · CLOVA Studio/HyperCLOVA X(Skill·RAG 브리프) · AiTEMS(추천) · Data Box(데이터 연동).
즉 자소서의 "HyperCLOVA X로 돌아간다"를 **구체적 스택 설계**로 격상한 것입니다.

### 10-5. 창작자 데이터(합성) 가정
`topics` 14개는 합성이며, 신호 정의는 — views(주간 조회수, 천) · growth(최근4주 vs 작년동기 %) · saturation(포화도 0~100, 높을수록 늦음) · opportunity(기회 점수 0~100) · completion(시청완료율) · remake(재생산 지수) · searchIn(검색 유입) · tagConv(상품태그 전환 %). 동일 데이터가 `data/synthetic-topics.csv`에도 있습니다.
